# v5 源代码与我的设计方案对比表

## 一、源码对应关系

| 分析内容 | v5 对应文件 | 核心函数/类 | 作用 |
|---|---|---|---|
| 出牌推荐入口 | `v5_rh/v5_rh/action_recommendation.py` | `recommend_card(...)` | 根据当前手牌、宝牌、弃牌、副露等信息，计算每张候选弃牌评分，并返回最高分弃牌。 |
| 动作推荐入口 | `v5_rh/v5_rh/action_recommendation.py` | `recommend_op(...)` | 判断是否执行吃、碰、杠、胡等操作，通过比较操作前后最高评分决定是否操作。 |
| 牌型评分调度 | `v5_rh/v5_rh/get_score_dict.py` | `get_score_dict(...)` | 同时计算平胡、十三烂、九幺、七对四种牌型的向听数，并选择接近最小向听数的牌型进入搜索。 |
| 平胡组合拆分 | `v5_rh/v5_rh/Node_PH.py` | `PingHu.pinghu_CS2()` | 去除宝牌后按花色拆分手牌，生成平胡候选组合，并计算向听数。 |
| 单花色递归拆分 | `v5_rh/v5_rh/Node_PH.py` | `tree_expand(...)`、`extract_32N(...)`、`get_32N(...)` | 枚举刻子、顺子、对子、搭子，并过滤不合理拆分。 |
| 字牌拆分 | `v5_rh/v5_rh/Node_PH.py` | `zi_expand(...)` | 字牌不能组成顺子，只枚举刻子、对子、孤张。 |
| 平胡搜索树 | `v5_rh/v5_rh/Node_PH.py` | `SearchTree_PH`、`Node_PH` | 根据拆分组合生成搜索树，扩展将牌、搭子、孤张，直到胡牌叶子节点。 |
| 搜索树扩展 | `v5_rh/v5_rh/Node_PH.py` | `SearchTree_PH.expand_node(...)` | 先定将，再扩展面子，结合宝牌处理宝吊、宝还原、补面子等情况。 |
| 胡牌路径评分 | `v5_rh/v5_rh/Node_PH.py` | `SearchTree_PH.cal_score(...)` | 根据番数、摸牌概率、进张权重计算胡牌路径分数。 |
| 弃牌分数回传 | `v5_rh/v5_rh/Node_PH.py` | `calculate_path_expectation(...)`、`get_discard_score(...)` | 搜索到胡牌叶子后，将路径收益累加到对应候选弃牌上。 |
| 麻将基础工具 | `v5_rh/v5_rh/lib_MJ.py`、`Utils.py` | `cal_xts_2(...)`、`t2tot3_info(...)`、`t1tot3_info(...)` 等 | 提供牌编码、向听数计算、搭子/孤张转面子的转换表、剩余牌统计等基础能力。 |

## 二、平胡组合拆分逻辑对比

| 对比维度 | v5 源代码平胡拆分逻辑 | 我设计的平胡拆分逻辑 | 分析 |
|---|---|---|---|
| 目标结构 | 以 `4 面子 + 1 将` 为平胡目标。 | 同样以 `4 面子 + 1 将` 为目标。 | 基础胡牌结构一致。 |
| 输入处理 | 输入手牌、副露、宝牌、飞宝等信息。 | 输入手牌、副露、红中数量、可选动作等信息。 | v5 面向上饶麻将；我的方案面向九江红中。 |
| 特殊牌处理 | 先移除宝牌，普通牌拆分完成后，在搜索树阶段处理宝牌。 | 先统计并移除红中，普通牌拆分后再用红中补缺。 | 两者都把特殊牌从普通拆分中分离出来，但特殊规则不同。 |
| 花色拆分 | 按万、条、筒、字牌拆分。 | 同样按万、条、筒、字牌拆分。 | 花色拆分可以减少无效组合枚举。 |
| 数牌处理 | 万、条、筒可拆刻子、顺子、对子、搭子。 | 万、条、筒可拆刻子、顺子、对子、搭子，并细分搭子质量。 | 我的方案会进一步区分两面、嵌张、边张价值。 |
| 字牌处理 | 字牌不能成顺子，只拆刻子、对子、孤张。 | 字牌同样不能成顺子，只拆刻子、对子、孤张。 | 字牌规则一致。 |
| 递归方式 | 通过 `extract_32N(...)` 递归抽取 3N、2N 组合。 | 通过计数表递归枚举刻子、顺子、对子、搭子、孤张。 | 思想一致，具体实现可以更模块化。 |
| 组合结构 | `[kz, sz, aa, ab, xts, leftCards]`。 | `面子集合、将牌候选、搭子集合、孤张集合、红中补缺信息、向听数`。 | 我的结构会额外记录红中使用情况，方便九江红中评估。 |
| 向听数计算 | 调用 `cal_xts_2(...)`，结合副露和宝牌数量计算。 | 根据完成面子数、将牌状态、搭子数、红中补缺能力计算。 | 九江红中需要把红中作为万能补牌资源纳入向听数修正。 |
| 不合理拆分过滤 | 会过滤如 `345` 被拆成 `35 + 4` 这类明显不合理拆法。 | 也需要过滤低质量拆法，并按搭子质量、孤张价值进一步排序。 | v5 已有基础过滤，我的方案会增加更细的评分标准。 |
| 组合排序 | 主要按刻子、顺子、对子数量排序，再结合向听数。 | 按向听数、面子数、搭子质量、孤张数量、红中消耗、剩余有效牌数排序。 | 我的方案更适合后续做稳定剪枝。 |
| 剪枝策略 | 通过过滤不合理拆分、只搜索接近最小向听数牌型减少计算。 | 只保留每类牌型下排名靠前的拆分组合，并限制红中补缺分支。 | 我的方案初期更强调可控计算量。 |
| 迁移难度 | 与上饶麻将宝牌规则耦合较强。 | 按九江红中规则设计，方便后续直接开发。 | v5 可参考结构，但不能直接照搬规则。 |

## 三、平胡组合拆分优缺点对比

| 方案 | 优点 | 缺点 |
|---|---|---|
| v5 源代码方案 | 拆分逻辑完整；能覆盖刻子、顺子、对子、搭子、孤张；按花色拆分效率较高；已经包含不合理组合过滤；能与搜索树和多牌型评分系统衔接。 | 代码与上饶麻将宝牌规则绑定较强；搭子质量区分不够细；组合数量多时计算压力较大；迁移到九江红中需要重写红中相关逻辑。 |
| 我的设计方案 | 普通牌和红中分开处理，逻辑更清晰；能区分两面、嵌张、边张等搭子价值；记录红中消耗，避免万能牌被低价值使用；更适合九江红中不可吃、红中不能碰杠、四红中胡牌等规则。 | 初期需要自己实现和测试；评分权重需要通过自博弈调整；红中补缺会增加搜索分支，需要剪枝控制；对手风险建模初期可能较弱。 |

## 四、平胡博弈树扩展评估逻辑对比

| 对比维度 | v5 源代码博弈树逻辑 | 我设计的博弈树逻辑 | 分析 |
|---|---|---|---|
| 搜索起点 | 从平胡拆分组合生成根节点。 | 从当前手牌枚举候选弃牌，也可结合最优拆分组合生成节点。 | v5 更偏“组合到胡牌路径”；我的方案更偏“弃牌到后续局面”。 |
| 节点内容 | 节点保存 `AAA`、`ABC`、`jiang`、`T2`、`T1`、`taking_set`、`king_num` 等信息。 | 节点保存当前手牌、已完成组合、候选弃牌、有效进张、红中使用情况、局面评分等信息。 | 我的节点会加入九江红中特有的红中状态。 |
| 扩展顺序 | 先扩展将牌，再判断胡牌，再扩展搭子和孤张。 | 先枚举候选弃牌，再枚举可能摸牌，必要时继续扩展下一轮弃牌/摸牌。 | 两种方式都能评估后续收益，只是搜索方向不同。 |
| 将牌处理 | 宝牌可作将，已有对子可作将，也可从孤张或搭子扩展成将。 | 普通对子可作将，红中可补将，两个红中也可作将。 | 九江红中需要重点处理红中补将。 |
| 面子扩展 | 使用 `t2tot3_dict`、`t1tot3_dict` 将搭子、孤张扩展成面子。 | 使用有效进张表，将搭子、孤张、对子结合红中补成面子或将。 | 可以参考 v5 的预计算表思想。 |
| 特殊牌扩展 | 处理宝吊、宝还原、飞宝等上饶麻将规则。 | 处理红中万能补牌、四红中胡、红中不能碰杠、跑红中翻倍等规则。 | 规则模块需要替换。 |
| 评分核心 | `路径分 = 番数 × 摸牌概率 × 进张权重`。 | `弃牌分 = 向听数分 + 有效进张分 + 组合结构分 + 红中利用分 + 规则收益分 - 惩罚项`。 | v5 更像胡牌路径期望；我的方案更像局面综合评分。 |
| 分数回传 | 搜索到胡牌叶子节点后，将路径分数回传给对应弃牌。 | 每个候选弃牌都直接计算后续若干层局面收益，再选最高分。 | 我的方案更容易实现初版，v5 后期上限更高。 |
| 概率来源 | 使用剩余牌数量计算 `T_SELFMO`。 | 同样使用剩余牌数量计算摸牌概率，并结合有效牌剩余张数。 | 可直接参考 v5 的剩余牌统计方式。 |
| 番数/收益 | 考虑碰碰胡、飞宝、宝还原、清一色等。 | 考虑九江红中胡牌、自摸、杠开、抢杠胡、跑红中翻倍、扎码等可扩展收益。 | 九江红中结算规则需要重新建模。 |
| 操作决策 | 操作前后最高评分对比，超过 5% 才推荐吃碰。杠倾向直接执行。 | 胡优先；不可吃；碰、杠通过操作前后评分和规则收益判断；红中禁止碰杠。 | v5 思路可复用，但操作集合要裁剪。 |
| 计算复杂度 | 搜索到胡牌路径，路径较完整但复杂。 | 初期可做两层或三层搜索，便于控制复杂度。 | 先实现稳定初版，再逐步加深搜索。 |

## 五、平胡博弈树评估优缺点对比

| 方案 | 优点 | 缺点 |
|---|---|---|
| v5 源代码方案 | 从胡牌路径反推弃牌价值，搜索逻辑较完整；结合番数、概率和进张权重；预计算转换表提高效率；能与多牌型评分合并。 | 搜索树复杂，调试难度较高；与上饶麻将宝牌规则耦合；对九江红中特殊规则不适配；对手风险建模较弱。 |
| 我的设计方案 | 搜索方向更直观，适合先实现九江红中初版；评分项可解释性强；容易加入红中万能牌、不可吃、红中不能碰杠等规则；计算深度可控。 | 不一定像 v5 那样完整搜索到胡牌终点；评分权重需要调试；如果只做浅层搜索，可能漏掉长期收益较高的路线。 |

## 六、已有代码问题与优化方案

| 问题方向 | 现有代码表现 | 优化方案 |
|---|---|---|
| 组合数量过多 | 递归枚举可能产生大量候选拆分，后续搜索压力较大。 | 增加组合排序和剪枝，只保留向听数低、面子多、搭子质量高、孤张少的前 N 个组合。 |
| 搭子价值区分不细 | 两面、嵌张、边张都作为搭子处理，价值差异不够明显。 | 给搭子设置权重：两面搭子 > 嵌张搭子 > 边张搭子。 |
| 孤张价值区分不足 | 孤张通常只是作为剩余牌处理。 | 按牌值区分孤张价值：中张数牌优于边张数牌，字牌孤张价值较低。 |
| 特殊牌规则耦合 | v5 中宝牌逻辑较多，直接迁移不适合九江红中。 | 抽象特殊牌模块：上饶使用宝牌策略，九江红中使用红中万能牌策略。 |
| 动作分支不适配 | v5 支持吃牌判断，而九江红中不可吃。 | 删除吃牌分支，只保留胡、碰、杠、过、出牌、听等合法动作。 |
| 红中处理缺失 | v5 没有九江红中的红中万能牌规则。 | 拆分阶段单独统计红中，评估阶段计算红中补将、补顺子、补刻子的价值。 |
| 红中碰杠限制 | v5 的宝牌规则不能直接表达红中不能碰杠。 | 动作判断时若操作牌是红中，直接排除碰杠候选。 |
| 四红中特判 | v5 没有四红中直接胡逻辑。 | 胡牌判断入口优先检查红中数量，四红中直接返回胡牌动作。 |
| 评分标准不统一 | 不同牌型评分方式不同，合并时可能存在尺度差异。 | 统一总评分框架：向听数、有效进张、剩余概率、牌型收益、风险惩罚。 |
| 对手风险考虑不足 | v5 主要偏自摸概率和自身牌型收益。 | 后续加入危险牌评估：根据对手弃牌、副露、听牌阶段降低高风险弃牌评分。 |

## 八、汇报时可以这样总结

本周主要完成了 v5 上饶麻将 AI 代码中平胡组合拆分逻辑和博弈树扩展评估逻辑的阅读分析。v5 的核心流程是先对手牌进行组合拆分，计算各牌型向听数，再选择接近最小向听数的牌型进入搜索树，通过摸牌概率、进张权重和番数计算胡牌路径收益，最后将收益回传到候选弃牌上，选择评分最高的牌作为推荐出牌。

在理解 v5 的基础上，我设计了一套适合九江红中麻将的平胡组合拆分思路：先将红中从普通手牌中单独分离，普通牌按花色递归拆分为刻子、顺子、对子、搭子和孤张，再利用红中作为万能牌补将、补顺子或补刻子，并根据向听数、搭子质量、孤张数量、红中消耗和有效进张数量对组合进行排序。

同时，我设计了一套平胡博弈树扩展评估逻辑：从候选弃牌出发，枚举打出每张牌后的局面，再扩展下一次可能摸到的牌，通过向听数变化、有效进张、组合结构、红中利用价值和规则收益综合评分。相比 v5 从胡牌叶子节点回传分数的方式，我的方案更适合九江红中 AI 初版开发，结构更直观，计算量也更容易控制。

后续优化重点是：细化各牌型组合拆分的评分标准，区分两面、嵌张、边张和孤张价值；将红中万能牌作为独立资源管理；删除九江红中不支持的吃牌分支；加入红中不能碰杠、四红中直接胡等特殊规则；最终参考 v5 的整体框架开发九江红中麻将 AI 并进行自博弈测试。
