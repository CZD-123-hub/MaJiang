## 二、v5 源代码整体决策流程

v5 代码中，外部调用入口主要在 `action_recommendation.py` 中，包括两个核心接口：

- `recommend_card(...)`：推荐当前应该打出的牌。
- `recommend_op(...)`：推荐是否执行吃、碰、杠、胡等动作。

### 1. 出牌决策流程

`recommend_card(...)` 的整体流程可以概括为：

```text
输入当前手牌、副露、宝牌、弃牌、剩余牌数等信息
-> 统计每种牌的剩余数量 LEFT_NUM
-> 计算每种牌的自摸概率 T_SELFMO
-> 调用 get_score_dict(...)
-> 分别计算平胡、十三烂、九幺、七对的向听数
-> 选择向听数较小的牌型进入搜索评估
-> 得到每张候选弃牌的评分
-> 选择评分最高的牌作为推荐出牌
```

其中 `get_score_dict.py` 是出牌评分的调度中心。它会创建四种牌型对象：

```python
PH = PingHu(...)
SSL = ShiSanLan(...)
JY = JiuYao(...)
QD = Qidui(...)
```

然后分别计算四种牌型的组合与向听数：

```python
CS_PH = PH.pinghu_CS2()
CS_SSL = SSL.ssl_CS()
CS_JY = JY.yaojiu_CS()
CS_QD = QD.qidui_CS()
```

之后取四种牌型中的最小向听数 `min_xts`。v5 并不会把所有牌型都完整搜索，而是只搜索“最小向听数”和“接近最小向听数”的牌型：

```text
如果某牌型 xts - 1 <= min_xts，则进入搜索
```

这样做的目的是减少搜索量，同时保留那些虽然当前向听数略高、但后续收益可能较好的牌型。

### 2. 动作决策流程

`recommend_op(...)` 用于判断是否执行吃、碰、杠等操作。它的基本思想是：

```text
先计算操作前手牌的最高评分
-> 枚举当前可执行的吃、碰、杠操作
-> 模拟操作后的手牌和副露
-> 再计算操作后的最高评分
-> 如果操作后评分明显高于操作前，则推荐执行该操作
```

源码中使用的判断条件大致是：

```python
if max_score > score_pre * 1.05:
    return 操作集合
```

也就是说，操作后的最高评分至少要比操作前高 5%，才会推荐吃碰。杠的处理更直接：暗杠、补杠、明杠在代码中倾向于直接执行。

这一点对下周开发九江红中很重要，因为九江红中的规则是**不可吃、可碰、可杠**，所以迁移时需要去掉吃牌分支，重点保留碰、杠、胡和出牌决策。

## 三、完整理清上饶麻将出牌决策和调用决策逻辑

上饶麻将 v5 代码的决策逻辑可以分成两条主线：一条是自己回合的**出牌决策**，另一条是别人出牌或自己摸牌后触发的**动作调用决策**。出牌决策关注“当前应该打哪张牌”，动作调用决策关注“当前是否应该吃、碰、杠、胡”。

### 1. 出牌决策入口

出牌决策入口是：

```python
recommend_card(...)
```

它位于 `action_recommendation.py` 中。该函数接收当前手牌 `cards`、副露 `suits`、宝牌 `king_card`、弃牌信息 `discards`、场面副露 `discards_op`、飞宝数量 `fei_king`、剩余牌数 `remain_num` 等信息。

它的主要任务不是直接通过规则判断某张牌能不能打，而是通过评分系统计算每张候选弃牌的价值，最后选择评分最高的一张牌。

完整流程如下：

```text
recommend_card(...)
-> 更新当前轮数、宝牌等全局信息
-> 根据弃牌、副露、自己手牌统计每种牌剩余数量 LEFT_NUM
-> 根据剩余牌数量计算自摸概率 T_SELFMO
-> 调用 get_score_dict(...)
-> 得到每张候选弃牌的评分 score_dict
-> 选择 score_dict 中评分最高的牌
-> 返回推荐出牌
```

其中，`LEFT_NUM` 表示每种牌理论上还剩几张，`T_SELFMO` 表示下一次自摸摸到某种牌的概率。v5 的出牌决策不是单纯看当前手牌形状，而是会结合剩余牌数量和摸牌概率。

### 2. 牌型评分调度

出牌决策的核心调度函数是：

```python
get_score_dict(...)
```

它位于 `get_score_dict.py` 中。这个函数会同时考虑四种牌型：

```text
平胡
十三烂
九幺
七对
```

对应代码中会创建四个牌型对象：

```python
PH = PingHu(...)
SSL = ShiSanLan(...)
JY = JiuYao(...)
QD = Qidui(...)
```

然后分别计算它们的组合和向听数：

```text
PH.pinghu_CS2()    -> 平胡组合和向听数
SSL.ssl_CS()       -> 十三烂组合和向听数
JY.yaojiu_CS()     -> 九幺组合和向听数
QD.qidui_CS()      -> 七对组合和向听数
```

之后取四种牌型中的最小向听数 `min_xts`。v5 不会对所有牌型都做完整搜索，而是只搜索当前最有希望的牌型：

```text
如果某牌型 xts - 1 <= min_xts，则该牌型进入搜索评估
```

这样做的原因是：如果某个牌型向听数明显落后，继续搜索它的收益通常不高，反而会增加计算量。保留“最小向听数”和“接近最小向听数”的牌型，可以在准确性和效率之间取得平衡。

### 3. 各牌型弃牌评分

进入搜索的牌型会分别计算弃牌评分：

```text
平胡 -> SearchTree_PH.get_discard_score()
十三烂 -> SSL.get_discard_score()
九幺 -> JY.get_discard_score()
七对 -> QD.get_discard_score()
```

每个牌型都会返回一个字典：

```text
{
    候选弃牌1: 评分,
    候选弃牌2: 评分,
    ...
}
```

然后 `get_score_dict(...)` 会把不同牌型中同一张弃牌的评分累加起来。也就是说，如果某张牌在平胡路线中值得打，在七对或其他牌型路线中也有价值，那么它的总评分会更高。

最终出牌逻辑可以理解为：

```text
一张牌的最终价值 = 它在所有候选牌型路线中的收益之和
```

这说明 v5 的出牌决策不是单一牌型决策，而是多牌型综合评分。

### 4. 平胡出牌评分内部逻辑

以平胡为例，平胡路线会进入：

```python
SearchTree_PH.get_discard_score()
```

其内部流程是：

```text
根据 PingHu.pinghu_CS2() 得到的组合生成搜索树根节点
-> 从根节点开始扩展将牌、搭子、孤张
-> 递归扩展到胡牌叶子节点
-> 对胡牌路径计算分数
-> 将路径分数回传给路径中可以丢弃的牌
-> 汇总每张弃牌得分
```

所以平胡的弃牌评分并不是只看“打出后向听数是多少”，而是会进一步估计这张牌打出后，后续有哪些可能的胡牌路径、这些路径的摸牌概率和番数收益如何。

### 5. 动作调用决策入口

动作调用决策入口是：

```python
recommend_op(...)
```

它同样位于 `action_recommendation.py` 中。该函数用于判断当前是否要吃、碰、杠、胡。

输入中比较关键的字段包括：

- `op_card`：别人打出的牌或当前可操作牌。
- `cards`：自己当前手牌。
- `suits`：自己的副露。
- `king_card`：宝牌。
- `canchi`：是否允许吃。
- `self_turn`：是否自己回合。
- `isHu`：当前是否可以胡。

### 6. 胡牌优先逻辑

动作决策最先判断能否胡牌：

```python
if isHu:
    return [], True
```

也就是说，如果外部规则已经判断当前可以胡，v5 会直接返回胡牌，不再继续比较吃碰杠收益。

### 7. 操作前评分

如果不能直接胡，代码会先计算“不操作”的价值。

流程如下：

```text
复制当前手牌 cards_pre
-> 调用 get_score_dict(cards_pre, suits, ...)
-> 得到操作前各候选弃牌评分
-> 取其中最大值作为 score_pre
```

这里的 `score_pre` 可以理解为：

```text
如果我现在不吃、不碰、不杠，继续保持当前手牌结构，下一步最优出牌的价值是多少。
```

后续所有操作都要和这个基准分进行比较。

### 8. 自己回合的杠牌判断

如果是自己回合，即 `self_turn=True`，代码主要判断暗杠和补杠：

```text
如果手牌中某张牌有 4 张 -> 推荐暗杠
如果副露中已有碰牌，手牌又摸到同一张 -> 推荐补杠
```

v5 对杠牌比较激进，满足暗杠或补杠条件时倾向于直接返回杠牌操作，而不是像吃碰一样严格做 5% 收益比较。

### 9. 其他玩家回合的操作判断

如果不是自己回合，即别人打出一张牌，代码会判断明杠、吃、碰。

流程如下：

```text
如果手牌中有 3 张 op_card -> 可以明杠，直接返回杠
否则枚举可吃、可碰组合
```

如果允许吃 `canchi=True`，并且 `op_card` 是数牌，代码会枚举三种可能吃法：

```text
[op_card - 2, op_card - 1]
[op_card - 1, op_card + 1]
[op_card + 1, op_card + 2]
```

如果手牌中存在对应搭子，则加入可操作集合。

碰牌判断比较简单：

```text
如果手牌中 op_card 数量 >= 2，则可以碰
```

### 10. 操作后评分

对于每一种候选操作，v5 会模拟操作后的局面：

```text
从手牌中移除用于吃/碰的牌
-> 把操作组合加入副露 suits
-> 调用 get_score_dict(...) 重新计算操作后的最高出牌评分
-> 记录操作后的最高评分
```

也就是说，代码并不是看到能吃、能碰就一定操作，而是会判断操作后整个手牌结构是否变好。

### 11. 操作执行条件

最后，v5 会比较操作前后的评分：

```text
如果 max_score > score_pre * 1.05
则执行评分最高的操作
否则选择过
```

这里的 `1.05` 表示操作后的收益至少要比操作前高 5%，才认为值得操作。这样可以避免因为评分微小波动导致频繁吃碰，也能减少破坏手牌整体结构的低价值操作。

### 12. 上饶麻将决策逻辑总结

上饶麻将 v5 的决策逻辑可以总结为：

```text
出牌决策：
当前手牌
-> 多牌型组合拆分
-> 计算各牌型向听数
-> 选择接近最小向听数的牌型
-> 各牌型分别搜索并计算弃牌评分
-> 合并不同牌型评分
-> 打出总评分最高的牌

动作决策：
当前可操作牌
-> 胡牌优先
-> 计算不操作时的最高评分
-> 枚举可吃、可碰、可杠操作
-> 模拟操作后局面
-> 重新计算操作后最高评分
-> 如果操作后收益明显提升，则执行操作
-> 否则过
```

整体来看，v5 的决策方式不是简单硬编码“有碰就碰、有杠就杠”，而是以向听数、搜索树、摸牌概率和牌型收益为基础，通过评分比较来选择动作。出牌决策体现为“多牌型综合评分”，动作调用决策体现为“操作前后收益比较”。

## 四、v5 平胡组合拆分逻辑

v5 的平胡逻辑主要在 `Node_PH.py` 的 `PingHu` 类中，核心函数是：

```python
PingHu.pinghu_CS2()
```

它的作用是把当前手牌拆成平胡相关的组合结构，并计算每种组合的向听数。

### 1. v5 的组合结构

v5 中每一种平胡拆分组合大致使用如下结构：

```text
[kz, sz, aa, ab, xts, leftCards]
```

各字段含义如下：

- `kz`：刻子集合，例如 `[3万, 3万, 3万]`。
- `sz`：顺子集合，例如 `[2万, 3万, 4万]`。
- `aa`：对子集合，可以作为将牌候选。
- `ab`：搭子集合，包括两面、边张、嵌张等。
- `xts`：该组合对应的向听数。
- `leftCards`：剩余孤张或暂时无法归入有效组合的牌。

这个结构的核心思想是：先把手牌拆成“已经完成的面子、可能成为将的对子、还差一张的搭子、暂时无用的孤张”，再根据这些信息判断距离平胡还差多少。

### 2. v5 的平胡拆分步骤

`pinghu_CS2()` 的主要流程如下：

```text
输入手牌
-> 去除宝牌 kingCard
-> 按花色拆分为万、条、筒、字
-> 万、条、筒分别调用 tree_expand(...)
-> 字牌调用 zi_expand(...)
-> 将四种花色的拆分结果组合起来
-> 处理 padding 和 t1
-> 根据剩余牌数量过滤无效搭子
-> 调用 cal_xts_2(...) 计算向听数
-> 返回所有候选组合
```

其中，`tree_expand(...)` 负责单一花色的拆分，内部又依赖：

- `get_32N(...)`：枚举当前花色中所有可能的 3N 和 2N 组合。
- `extract_32N(...)`：递归抽取 3N、2N 组合。
- `get_effective_cards(...)`：计算搭子的有效进张。

### 3. v5 的递归拆分思想

v5 的拆分方式是递归枚举。每一次递归都会从当前牌中找出所有可行的 3N 或 2N：

- 刻子 AAA。
- 顺子 ABC。
- 对子 AA。
- 连续搭子 AB。
- 嵌张搭子 AC。

然后选择其中一种组合，从手牌中移除，再对剩余牌继续递归。直到没有可拆的 3N 或 2N，就把当前拆分路径保存下来。

这种方法的优点是比较全面，能够保留多种可能拆法；缺点是组合数量可能很多，需要后续排序、去重和过滤。

### 4. v5 的不合理组合过滤

v5 在 `tree_expand(...)` 中会过滤一些不合理拆法。例如一组 `3、4、5`，如果被拆成 `3、5` 搭子加 `4` 孤张，这种拆法虽然形式上存在，但实际不如直接拆成顺子 `3、4、5`。因此代码会判断孤张是否仍然是某个搭子的有效牌，如果是，就认为这次拆分不合理并丢弃。

这个过滤思路是合理的，因为它减少了明显劣质的组合，也能降低后续搜索树的计算量。

### 5. v5 对宝牌的处理

v5 是上饶麻将代码，里面有宝牌、飞宝、宝还原、宝吊等规则。因此 `pinghu_CS2()` 会先把宝牌从手牌中移除，不让宝牌参与普通牌自然拆分。后续在搜索树扩展时，再根据宝牌数量判断宝牌如何补将、补面子或飞宝。

这种设计把“普通牌自然结构”和“宝牌万能作用”分开处理，逻辑比较清晰。

## 五、我设计的平胡牌型组合拆分逻辑

我设计的平胡组合拆分逻辑，目标是服务于后续九江红中麻将 AI。九江红中与上饶麻将不同，它没有宝牌和飞宝，但有红中万能牌。红中可以在胡牌时替代任意牌，但红中不能碰杠，四红中可直接胡。

因此，我的设计思路是：**普通牌先自然拆分，红中单独作为万能资源参与补缺。**

### 1. 拆分目标

平胡目标仍然是：

```text
4 组面子 + 1 组将牌
```

其中：

- 面子可以是顺子或刻子。
- 将牌是对子。
- 红中可以在胡牌判断和组合评估时补充缺失的牌。

### 2. 具体拆分流程

我的组合拆分流程如下：

```text
输入手牌
-> 统计红中数量 hongzhong_num
-> 从手牌中移除红中
-> 普通牌按万、条、筒、字分组
-> 万、条、筒递归拆分刻子、顺子、对子、搭子、孤张
-> 字牌只拆刻子、对子、孤张
-> 合并各花色拆分结果
-> 对每种组合计算普通向听数
-> 再根据红中数量修正向听数和组合价值
-> 按综合评分排序，保留较优组合
```

### 3. 普通牌递归拆分规则

对于普通牌，递归时按以下优先级尝试：

1. 如果某张牌有 3 张，尝试拆成刻子。
2. 如果同花色存在连续三张，尝试拆成顺子。
3. 如果某张牌有 2 张，尝试拆成对子。
4. 如果同花色存在连续两张，尝试拆成两面或边张搭子。
5. 如果同花色存在隔一张的两张，尝试拆成嵌张搭子。
6. 如果都不能组成有效结构，则作为孤张。

需要注意的是，这里的“优先级”不是只保留第一种拆法，而是用于生成候选分支时排序。因为同一副手牌可能存在多种合理拆法，不能过早只选一种。

### 4. 红中的补牌规则

红中不直接参与普通牌拆分，而是在组合生成后用于补缺：

1. **补将牌**
   如果当前没有对子，可以用 1 张红中和 1 张普通牌组成将牌；也可以用 2 张红中直接作为将。

2. **补刻子**
   如果有对子，可以用 1 张红中补成刻子；如果有单张，也可以用 2 张红中补成刻子。

3. **补顺子**
   如果有搭子，例如 `2万 3万`，可以用红中补成 `1万 2万 3万` 或 `2万 3万 4万`。如果是嵌张搭子，例如 `2万 4万`，可以用红中补成 `2万 3万 4万`。

4. **四红中特殊胡**
   如果手牌中有 4 张红中，则单独判断四红中直接胡，不进入普通平胡拆分流程。

### 5. 组合排序规则

每种拆分组合都需要评分排序。我的排序标准是：

```text
向听数越小越好
完整面子越多越好
对子越稳定越好
有效搭子越多越好
两面搭子优先于嵌张，嵌张优先于边张
孤张越少越好
消耗红中越少越好
有效进张剩余数量越多越好
```

这里特别加入“红中消耗数量”作为排序因素，因为红中是万能资源，过早消耗红中补低价值结构，可能会降低后续灵活性。

### 6. 我的拆分逻辑优点

我的设计有几个优点：

1. 普通牌和红中分开处理，逻辑清晰。
2. 能保留多种拆分结果，不会过早丢掉可能更优的牌型。
3. 能区分两面、边张、嵌张等搭子价值。
4. 可以根据九江红中特殊规则进行扩展。
5. 后续可以同时支持平胡、七对、碰碰胡等牌型的统一评估。

### 7. 我的拆分逻辑可能不足

这个方案也有不足：

1. 如果保留组合太多，计算量会增加。
2. 红中补牌方式较多，搜索分支容易变大。
3. 初期评分规则主要依靠经验，需要通过自博弈结果调整权重。
4. 如果只考虑自己的组合，不考虑对手危险牌，实际对局中可能偏进攻。

## 七、v5 平胡博弈树扩展评估逻辑

v5 的平胡搜索树主要由 `SearchTree_PH` 和 `Node_PH` 实现。

### 1. 搜索树节点结构

`Node_PH` 节点中保存了当前扩展状态，主要字段包括：

- `AAA`：已经形成的刻子。
- `ABC`：已经形成的顺子。
- `jiang`：将牌。
- `T2`：搭子集合。
- `T1`：孤张集合。
- `raw`：当前待扩展集合。
- `taking_set`：到达胡牌路径需要摸到的牌。
- `taking_set_w`：进张权重。
- `king_num`：剩余宝牌数量。
- `fei_king`：飞宝数量。
- `children`：子节点。

可以理解为，一个节点表示：

```text
当前已经完成了哪些面子和将牌，还剩哪些搭子、孤张需要继续扩展。
```

### 2. 搜索树生成流程

`SearchTree_PH.generate_tree()` 会根据平胡组合拆分结果生成根节点。每个根节点包含：

- 当前拆分中已经完成的刻子和顺子。
- 副露中的刻子和顺子。
- 当前对子和搭子。
- 当前孤张。
- 宝牌数量。

然后从每个根节点开始调用：

```python
expand_node(node)
```

递归扩展整棵搜索树。

### 3. v5 的扩展顺序

`expand_node(...)` 的扩展顺序非常明确：

```text
先扩展将牌
-> 判断是否已经胡牌
-> 再扩展面子
```

#### 第一步：扩展将牌

v5 会依次考虑：

1. 两张宝牌直接作为将牌。
2. 一张宝牌进行宝吊。
3. 已有对子作为将牌。
4. 没有对子时，从孤张或搭子中选择牌扩展成将。

#### 第二步：判断胡牌

如果当前已经有 4 个面子，并且已经有将牌，就认为这条路径到达胡牌叶子节点。

#### 第三步：扩展面子

如果还没有胡牌，则继续扩展：

1. 优先把 `T2` 搭子扩展成 `T3` 面子。
2. 如果没有可扩展搭子，再把 `T1` 孤张扩展成 `T3` 面子。

v5 使用 `GameConfig` 中预先生成的转换表：

- `t2tot3_dict`：搭子变成面子的转换表。
- `t1tot3_dict`：孤张变成面子的转换表。

这样搜索时不需要临时计算每个搭子或孤张能变成哪些面子，提高了效率。

### 4. v5 的评分逻辑

v5 的节点评分函数是：

```python
cal_score(node)
```

核心思想可以概括为：

```text
节点分数 = 番数 × 摸牌概率 × 进张权重
```

其中：

- 摸牌概率来自 `T_SELFMO`。
- 进张权重来自 `taking_set_w`。
- 番数由 `Fan_PH.fanDetect()` 计算。

番数中会考虑：

- 碰碰胡。
- 飞宝。
- 宝还原。
- 清一色。
- 单吊等情况。

### 5. 从胡牌路径反推弃牌评分

v5 并不是简单地对当前弃牌评分，而是先搜索到胡牌叶子节点，再把胡牌路径的收益回传给相关弃牌。

核心函数是：

```python
calculate_path_expectation(node)
```

它的流程是：

```text
深度搜索到胡牌叶子节点
-> 计算该胡牌路径分数
-> 找出这条路径中可丢弃的 T1、T2、飞宝等
-> 把路径分数累加到对应弃牌上
```

最后 `get_discard_score()` 会汇总所有路径：

```text
某张弃牌总分 = 所有保留该弃牌收益路径的分数之和
```

最终得到每张候选弃牌的评分。

## 八、我设计的平胡博弈树扩展评估逻辑

我设计的博弈树评估逻辑，目标是先保证九江红中 AI 能稳定出牌，再逐步增加搜索深度和规则细节。

### 1. 基本思路

我的搜索树可以分成两种阶段：

```text
候选弃牌评估
后续摸牌扩展
```

根节点是当前手牌。如果是自己摸牌后，需要从 14 张牌中选择一张打出。

### 2. 第一层：枚举候选弃牌

对于当前手牌中的每一种不同牌，分别模拟打出：

```text
当前 14 张手牌
-> 打出候选牌
-> 得到 13 张手牌
-> 重新计算平胡组合拆分和向听数
```

这一层保证所有可能出牌都会被考虑，不会因为局部优先级而漏掉候选。

### 3. 第二层：枚举可能摸牌

对于每个弃牌后的 13 张手牌，枚举下一次可能摸到的牌：

```text
弃牌后手牌
-> 枚举牌墙中仍可能摸到的牌
-> 加入手牌
-> 重新拆分
-> 判断向听数、有效进张、组合质量变化
```

如果摸到某张牌能降低向听数，或者能形成更好的搭子、对子、面子，则说明当前弃牌选择更有价值。

### 4. 后续多层扩展

如果计算量允许，可以继续扩展：

```text
弃牌 -> 摸牌 -> 再弃牌 -> 再摸牌
```

但完整搜索会导致节点数量快速增长，所以我设计的剪枝策略是：

1. 只扩展向听数较低的组合。
2. 只保留每个弃牌下评分最高的前若干种拆分。
3. 优先扩展有效进张多的路径。
4. 对已经明显落后的路径提前停止。
5. 对红中消耗过多但收益不明显的路径降权。

### 5. 评分公式设计

我设计的弃牌评分可以由几部分组成：

```text
弃牌得分 =
向听数基础分
+ 有效进张分
+ 剩余牌数量分
+ 组合结构分
+ 红中利用分
+ 规则收益分
- 孤张惩罚
- 红中过度消耗惩罚
- 危险牌惩罚
```

具体解释如下：

1. **向听数基础分**
   向听数越小，说明越接近胡牌，分数越高。

2. **有效进张分**
   能让向听数下降的牌越多，说明这张弃牌保留的牌型更灵活。

3. **剩余牌数量分**
   如果有效牌在牌墙中剩余数量多，实际摸到的概率更高。

4. **组合结构分**
   完整面子、稳定对子、好搭子越多，组合越好。

5. **红中利用分**
   如果红中能补关键缺口，例如补将、补嵌张、补刻子，则加分。

6. **规则收益分**
   根据九江红中特殊规则修正，例如自摸、杠开、抢杠胡、跑红中翻倍等。

7. **孤张惩罚**
   孤张越多，说明手牌结构越散，应该扣分。

8. **红中过度消耗惩罚**
   如果一个方案为了降低一点向听数消耗过多红中，但没有形成明显优势，应降低评分。

9. **危险牌惩罚**
   后续可以结合其他玩家弃牌和副露，降低危险弃牌的评分。

### 6. 九江红中特殊规则处理

九江红中迁移时，需要特别处理：

1. 不生成吃牌分支，因为九江红中不可吃。
2. 红中不能碰杠，所以动作判断时红中不进入碰杠候选。
3. 四红中直接胡，优先级高于普通平胡搜索。
4. 红中在胡牌时可代替任意牌，因此组合评估时要计算红中的补缺价值。
5. 如果启用“跑红中翻倍”，则打出红中会影响后续胡分，需要在评分中单独考虑。

## 十、从组合拆分角度分析已有代码问题

我负责的方向是：

```text
各牌型的组合拆分逻辑是否合理，进行优化。
```

从这个方向看，已有代码主要有以下问题。

### 1. 拆分组合数量可能过多

递归枚举会保留大量拆分结果。虽然这样比较全面，但如果全部进入搜索树，会导致计算量变大。

优化方案：

```text
按向听数、面子数、搭子质量、孤张数量进行排序，只保留前 N 个较优组合。
```

### 2. 搭子质量区分不够细

同样是搭子，两面搭子通常比嵌张和边张更好。例如：

- `3万 4万` 可以进 `2万` 或 `5万`，价值较高。
- `3万 5万` 只能进 `4万`，价值较低。
- `1万 2万` 只能进 `3万`，也比中张两面差。

优化方案：

```text
给不同搭子设置不同权重：
两面搭子 > 嵌张搭子 > 边张搭子。
```

### 3. 孤张价值没有充分区分

不同孤张的价值不同。中张孤张更容易形成搭子，边张和字牌孤张价值较低。

优化方案：

```text
孤张也设置价值：
中张数牌 > 边张数牌 > 字牌孤张。
```

### 4. 特殊牌型和特殊规则耦合较强

v5 主要围绕上饶麻将宝牌设计，九江红中的红中规则不能直接复用。

优化方案：

```text
把特殊牌处理抽象成独立模块：
上饶麻将使用宝牌策略；
九江红中使用红中万能牌策略。
```

### 5. 不同牌型之间缺少统一评分标准

平胡、七对、九幺、十三烂的组合拆分方式不同，如果最终评分标准不统一，可能导致不同牌型分数不可比。

优化方案：

```text
统一使用：
向听数 + 有效进张 + 剩余牌概率 + 牌型收益 + 风险修正
作为总评估框架。
```

### 6. 九江红中规则需要重新适配

九江红中有几个关键规则：

- 不可吃。
- 可碰、可杠。
- 红中胡牌时可替代任意牌。
- 红中不能碰杠。
- 四红中可直接胡。

优化方案：

```text
拆分阶段单独统计红中；
操作阶段禁止红中碰杠；
搜索阶段删除吃牌分支；
胡牌判断阶段加入四红中特判和红中补缺逻辑。
```
