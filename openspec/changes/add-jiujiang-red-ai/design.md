## Context

当前仓库中有两部分可参考内容：

- `v5_rh/`：上饶麻将 AI 代码，已经实现较完整的动作推荐、多牌型调度、平胡拆分和搜索树评分。
- `mahjong_ai/`：较小的教学/演示实现，包含基础牌工具、普通平胡拆分和一层弃牌评估。

九江红中麻将与上饶麻将规则差异较大：九江红中不可吃，红中在胡牌时可作为万能牌，红中不能碰杠，四红中可直接胡。v5 中宝牌、飞宝、宝吊、宝还原等逻辑不能直接套用。因此本次开发应新增独立模块，而不是改写上饶麻将代码。

## Goals / Non-Goals

**Goals:**

- 新建独立的九江红中 AI 包，明确拥有自己的规则边界。
- 提供等价于 `/get_action` 和 `/round_end` 的 Python 可调用入口。
- 支持从 `action_cards` 中选择合法动作：出牌、过、胡、碰、杠。
- 实现第一版带红中万能牌处理的平胡评估。
- 保持实现可测试、可解释，优先补充聚焦单元测试。
- 保持 `v5_rh/` 作为参考代码，不修改其行为。

**Non-Goals:**

- 第一版不实现完整 battle-server HTTP 服务。
- 第一版不完整模拟扎码、加买、所有房间选项的结算收益。
- 第一版不实现深度对手建模。
- 不修改 `v5_rh/`。
- 不实现吃牌，因为九江红中不允许吃。

## Decisions

### Decision 1: 新增独立 `jiujiang_ai` 包

本次开发新增独立包，而不是直接修改 `v5_rh` 或在现有通用 `mahjong_ai` 上硬扩展。

理由：

- `v5_rh` 与上饶麻将宝牌/飞宝规则耦合较强。
- `mahjong_ai` 可作为简化参考，但它仍包含非九江红中的字牌。
- 独立包可以让九江红中规则更清楚，也能降低迁移和调试风险。

备选方案：

- 直接修改 `v5_rh`。该方案被放弃，因为会把两套麻将规则混在一起，后续排查问题更困难。

### Decision 2: 牌编码沿用 v5 风格的数牌编码

使用 v5 风格的十六进制数牌编码：

- 万：`0x01`-`0x09`
- 条：`0x11`-`0x19`
- 筒：`0x21`-`0x29`
- 红中：`0x35`

理由：

- 现有输入样例和 v5 编码方式基本兼容。
- 保持该编码方便与 v5 对照和调试。
- 九江模块可以在校验层拒绝其他字牌。

### Decision 3: 红中作为单独的万能牌资源处理

平胡拆分时先把红中从普通手牌中分离出来，普通数牌先做自然拆分，再消耗红中去补对子、顺子或刻子。

理由：

- 符合红中在胡牌时可以替代任意牌的规则。
- 避免红中干扰普通顺子/刻子枚举。
- 便于在评分时惩罚低价值的红中过度消耗。

### Decision 4: 第一版弃牌评估采用浅层搜索

第一版采用可解释的浅层评估模型：

```text
候选弃牌
-> 弃牌后的 13 张手牌
-> 最优平胡向听数 / 有效进张估计
-> 根据向听数、有效牌、面子/搭子结构、红中使用、孤张数量评分
```

理由：

- 速度快，便于测试。
- 与当前项目中 `mahjong_ai/search_tree.py` 的简化搜索方向一致。
- 后续可以在动作合法性和接口稳定后，再引入更接近 v5 的深层胡牌路径评分。

### Decision 5: 动作 API 信任 `action_cards` 候选，但本地重新校验九江规则

服务端输入会通过 `data["action_cards"]` 给出可选动作。AI 只从候选动作中选择，但仍需要本地校验九江规则：

- 不选择吃。
- 不选择红中碰/杠。
- 可以胡时优先胡。
- 不支持或收益不明显时返回过。

理由：

- 输入说明中明确可操作类型来自 `action_cards`。
- 本地规则校验可以避免泛麻将输入中出现九江红中不支持的动作。

## Risks / Trade-offs

- [风险] 第一版浅层评分可能弱于 v5 搜索到胡牌叶子的期望值算法。→ 缓解：将评分项拆分清楚，后续可替换或叠加更深搜索。
- [风险] 红中万能牌向听数如果处理过松，可能过度乐观。→ 缓解：增加无红中、一张红中、多张红中、四红中等测试用例。
- [风险] 服务端可能传入未知或变体动作。→ 缓解：集中定义动作常量，未知动作默认过。
- [风险] 扎码、加买等结算选项未来会影响策略。→ 缓解：第一版不做完整结算，但评估器预留规则权重扩展点。

## Migration Plan

1. 新增 `jiujiang_ai/` 包和测试，不修改 `v5_rh/`。
2. 实现核心牌规则和动作输出格式。
3. 实现带红中支持的平胡分析。
4. 实现弃牌和操作选择。
5. 运行单元测试和简单样例调用。
6. 后续如学长测试环境需要，再增加 HTTP 包装或博弈接口集成。

本次变更是新增式的，回滚方式简单：删除 `jiujiang_ai/` 和对应测试/文档即可。
