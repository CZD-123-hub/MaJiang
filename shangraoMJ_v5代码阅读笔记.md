# shangraoMJ_v5 代码阅读笔记

## 一、项目整体作用

`v5_rh` 是一个上饶麻将 AI 决策代码，主要功能是：

1. 根据当前手牌、副露、宝牌、弃牌等信息，推荐出牌。
2. 根据别人打出的牌，判断是否吃、碰、杠、胡。
3. 同时考虑多种胡牌类型：平胡、十三烂、九幺、七对。
4. 使用“向听数 + 搜索树 + 概率期望值”的方式评估每种出牌。

核心思想可以概括为：

```text
先拆牌
-> 算各牌型向听数
-> 选择向听数较小的牌型进入搜索
-> 扩展到可能胡牌的路径
-> 用自摸概率、进张权重、番数计算期望值
-> 汇总每张弃牌的得分
-> 选择得分最高的牌作为推荐出牌
```

## 二、主要文件说明

### 1. `action_recommendation.py`

这是外部调用的入口文件。

主要函数：

- `recommend_card(...)`：推荐出牌。
- `recommend_op(...)`：推荐吃、碰、杠等操作。

`recommend_card` 的主要流程：

```text
输入当前手牌、宝牌、弃牌、副露等信息
-> 计算剩余牌数量 LEFT_NUM
-> 计算每张牌的自摸概率 T_SELFMO
-> 调用 get_score_dict(...)
-> 得到每张候选弃牌的评分
-> 返回评分最高的弃牌
```

其中：

- `LEFT_NUM` 表示每种牌理论上还剩几张。
- `T_SELFMO` 表示摸到每种牌的概率。
- `king_card` 是宝牌。
- `fei_king` 是飞宝数量。

### 2. `get_score_dict.py`

这是出牌评分的调度中心。

它会同时创建四种牌型对象：

```python
PH = PingHu(...)
SSL = ShiSanLan(...)
JY = JiuYao(...)
QD = Qidui(...)
```

对应四种牌型：

- `PingHu`：平胡。
- `ShiSanLan`：十三烂。
- `JiuYao`：九幺。
- `Qidui`：七对。

然后分别计算四种牌型的组合和向听数：

```python
CS_PH = PH.pinghu_CS2()
CS_SSL = SSL.ssl_CS()
CS_JY = JY.yaojiu_CS()
CS_QD = QD.qidui_CS()
```

之后得到向听数列表：

```python
xts_list = [平胡向听数, 十三烂向听数, 九幺向听数, 七对向听数]
```

代码不会把所有牌型都搜索到底，而是只搜索“最小向听数”和“最小向听数 + 1”范围内的牌型：

```text
如果某牌型 xts - 1 <= min_xts，就进入搜索
```

这样做是为了减少计算量，也是一种剪枝。

最后，它会把不同牌型下同一张弃牌的评分加起来：

```text
同一张弃牌如果在多个牌型中都有价值，则总评分相加
```

最终返回：

```python
score_dict, min_xts
```

其中 `score_dict` 的格式大概是：

```python
{
    0x01: 0.123,
    0x11: 0.456,
    ...
}
```

键是候选弃牌，值是该弃牌的评估分。

### 3. `lib_MJ.py`

这是麻将基础工具库，包含：

- 牌编码。
- 花色拆分。
- 搭子、顺子、刻子的枚举。
- 向听数计算。
- `t1 -> t3`、`t2 -> t3` 的转换表。

重要函数：

- `split_type_s(cards)`：按万、条、筒、字牌拆分。
- `get_32N(cards)`：找出当前牌里所有可能的 3N 和 2N。
- `extract_32N(...)`：递归枚举组合拆分。
- `tree_expand(cards)`：把递归拆分结果整理成统一结构。
- `get_t2info()`：生成所有搭子以及对应有效牌。
- `get_t3info()`：生成所有刻子和顺子。
- `t2tot3_info()`：生成搭子变成面子的方式。
- `t1tot3_info()`：生成孤张变成面子的方式。
- `cal_xts_2(...)`：计算向听数。

它采用的牌编码和我前两个任务中使用的编码是一致的：

```text
万子：0x01 - 0x09
条子：0x11 - 0x19
筒子：0x21 - 0x29
字牌：0x31 - 0x37
```

## 三、平胡流程重点

平胡是代码里最复杂、最核心的牌型，对应文件：

```text
Node_PH.py
```

主要类：

- `PingHu`：负责平胡组合拆分。
- `SearchTree_PH`：负责平胡搜索树扩展和评分。
- `Node_PH`：搜索树节点。
- `Fan_PH`：平胡番数计算。

### 1. 平胡组合拆分

入口函数是：

```python
PingHu.pinghu_CS2()
```

它的流程：

```text
去掉宝牌
-> 按花色拆分为万、条、筒、字
-> 每个花色单独递归拆分
-> 把不同花色的拆分结果组合起来
-> 过滤掉无效搭子
-> 计算每个组合的向听数
-> 返回组合集合
```

组合结构大致是：

```python
[kz, sz, aa, ab, xts, leftCards]
```

含义：

- `kz`：刻子集合。
- `sz`：顺子集合。
- `aa`：对子集合。
- `ab`：搭子集合，包括 AB、AC。
- `xts`：向听数。
- `leftCards`：剩余孤张。

这和任务一里要求的输出结构基本一致。

### 2. 平胡搜索树节点

`Node_PH` 节点保存当前搜索状态：

```python
take        本节点需要摸到的牌
AAA         已完成刻子
ABC         已完成顺子
jiang       将牌
T2          搭子集合
T1          孤张集合
raw         当前待扩展集合
taking_set  到达胡牌路径需要摸到的牌
king_num    剩余宝牌数
fei_king    飞宝数
children    子节点
```

可以理解为：

```text
一个 Node_PH 表示“当前已经凑到了哪些面子，还差哪些牌才能胡”
```

### 3. 平胡搜索树扩展

核心函数是：

```python
SearchTree_PH.expand_node(node)
```

扩展逻辑主要分三步。

第一步：先扩展将牌。

代码会依次考虑：

```text
两张宝牌直接当将
一张宝牌宝吊
已有对子当将
没有对子时，从孤张或搭子中选一张扩展为将
```

第二步：判断是否已经胡牌。

如果：

```text
已有 4 个面子，并且已有将牌
```

说明这条路径已经到达胡牌叶子节点。

第三步：扩展面子。

如果还没有 4 个面子，就继续把：

```text
T2 搭子扩展成 T3 面子
T1 孤张扩展成 T3 面子
```

它不是临时硬算，而是使用 `GameConfig` 里提前生成好的转换表：

```python
game_config.t2tot3_dict
game_config.t1tot3_dict
```

例如：

```text
[1W, 2W] 可以通过摸 3W 变成 [1W, 2W, 3W]
[1W] 可以通过摸 [2W, 3W] 变成 [1W, 2W, 3W]
```

### 4. 平胡节点评分

评分函数是：

```python
SearchTree_PH.cal_score(node)
```

它的核心公式可以理解为：

```text
节点分数 = 番数 × 摸牌概率 × 进张权重
```

其中：

- 摸牌概率来自 `game_config.T_SELFMO`。
- 进张权重来自 `taking_set_w`。
- 番数由 `Fan_PH.fanDetect()` 计算。

代码里会对以下情况加番：

- 碰碰胡。
- 飞宝。
- 宝还原。
- 清一色。
- 单吊。

### 5. 从胡牌路径反推弃牌评分

核心函数是：

```python
SearchTree_PH.calculate_path_expectation(node)
```

它会深度搜索到胡牌叶子节点，然后把这条路径的分数分配给路径中可以丢弃的牌。

逻辑是：

```text
找到胡牌叶子节点
-> 计算这条胡牌路径的分数
-> 找到这条路径中剩下的 T1、T2、飞宝等可丢弃牌
-> 把该路径分数累加到对应弃牌上
```

最后：

```python
get_discard_score()
```

会把所有路径分数汇总成：

```python
{
    弃牌1: 总分,
    弃牌2: 总分,
    ...
}
```

这就是平胡牌型下的推荐出牌评分。

## 四、其他牌型流程

### 1. 七对 `Node_Qidui.py`

七对的组合比较固定：

```text
已有对子越多，向听数越低
单张越少，越接近七对
宝牌可以补对子
```

主要函数：

- `qidui_CS()`：生成七对组合和向听数。
- `expand_node()`：把单张扩展成对子。
- `evaluate()`：搜索到 7 个对子后计算概率和番数。
- `get_discard_score()`：得到每张弃牌评分。

七对没有平胡那么多顺子、刻子拆分，因此搜索树更简单。

### 2. 十三烂 `Node_SSL.py`

十三烂要求同花色之间间隔较大，例如：

```text
1、4、7
1、4、9
2、5、8
```

主要函数：

- `ssl_CS()`：计算当前牌里哪些牌符合十三烂结构。
- `efc_ssl()`：计算后续需要摸哪些牌。
- `expand_node()`：扩展十三烂搜索树。
- `evaluate()`：计算路径分数。

十三烂的核心是判断每个花色中哪些牌能组成“烂牌结构”。

### 3. 九幺 `Node_JY.py`

九幺只关心幺九牌和字牌：

```python
[1, 9, 0x11, 0x19, 0x21, 0x29, 东, 南, 西, 北, 中, 发, 白]
```

主要函数：

- `yaojiu_CS()`：把手牌分成九幺有效牌和无效牌。
- `get_discard_score()`：根据剩余九幺牌概率计算出牌评分。

非九幺牌通常会被作为候选弃牌。

## 五、推荐出牌总流程

完整流程如下：

```text
recommend_card(...)
    |
    |-- 统计剩余牌 LEFT_NUM
    |-- 计算自摸概率 T_SELFMO
    |
    |-- get_score_dict(...)
            |
            |-- 平胡组合与向听数 PH.pinghu_CS2()
            |-- 十三烂组合与向听数 SSL.ssl_CS()
            |-- 九幺组合与向听数 JY.yaojiu_CS()
            |-- 七对组合与向听数 QD.qidui_CS()
            |
            |-- 找到最小向听数 min_xts
            |-- 只搜索 min_xts 附近的牌型
            |
            |-- 平胡搜索树 SearchTree_PH.get_discard_score()
            |-- 十三烂评分 SSL.get_discard_score()
            |-- 九幺评分 JY.get_discard_score()
            |-- 七对评分 QD.get_discard_score()
            |
            |-- 合并不同牌型下的弃牌评分
    |
    |-- 选择 score_dict 中评分最高的牌
    |
    |-- 返回推荐出牌
```

## 六、推荐操作流程

`recommend_op(...)` 用于判断吃、碰、杠。

它的思路是：

```text
先计算操作前的最高评分
-> 枚举可操作集合，比如吃、碰、杠
-> 模拟操作后的手牌和副露
-> 再计算操作后的最高评分
-> 如果操作后评分明显更高，就推荐操作
```

关键判断：

```python
if max_score > score_pre * 1.05:
    return 操作集合
```

也就是说，操作后评分至少要比操作前高 5%，才会推荐吃碰。

杠的处理更直接：

```text
如果能暗杠、补杠、明杠，代码倾向于直接推荐杠。
```

## 七、我对代码思想的理解

这份代码不是单纯用规则表直接判断“打哪张”，而是先把手牌拆成不同牌型组合，再把组合看作搜索树的起点。

平胡部分最典型：

```text
当前组合中已有的刻子、顺子、对子、搭子、孤张
-> 通过摸牌把搭子和孤张补成面子
-> 递归扩展到胡牌状态
-> 每条胡牌路径都有一个概率和番数
-> 哪张弃牌能保留更多高价值胡牌路径，就给哪张牌更高分
```

所以它本质上是：

```text
基于知识的牌型拆分
+
基于概率的树搜索评估
```

其中“知识”体现在：

- 上饶麻将牌型规则。
- 宝牌、飞宝、宝吊、宝还原。
- 平胡、七对、十三烂、九幺等牌型。
- 番数计算。

“树搜索”体现在：

- 从组合根节点开始扩展。
- 将 T2、T1 扩展成 T3。
- 递归直到胡牌叶子节点。
- 回溯累计弃牌评分。

“概率评估”体现在：

- 用剩余牌数量估算自摸概率。
- 用进张权重区分对子、搭子等价值。
- 用番数放大高价值胡牌路径。

## 八、代码中的剪枝和平衡

代码里有多处减少计算量的设计。

第一，牌型层面的剪枝：

```text
只搜索最小向听数及其附近的牌型
```

如果某个牌型向听数明显落后，就不再搜索。

第二，组合层面的过滤：

```text
拆分手牌时，会过滤不合理的组合划分
```

例如把 `3、4、5` 拆成 `3、5` 搭子加 `4` 孤张，这种情况会被认为不合理。

第三，转换表预计算：

```text
t1tot3_dict 和 t2tot3_dict 在 GameConfig 中提前生成
```

搜索时不用重复计算孤张、搭子如何变成面子。

第四，搜索结果去重：

```text
相同的 taking_set 只保留更高分的路径
```

这减少了重复状态对评分的影响。

平衡点是：

```text
不全量搜索所有牌型、所有深度、所有未来回合；
但对当前最有希望的牌型进行较完整的路径扩展。
```

这样既能体现搜索决策，又能避免计算量爆炸。

## 九、可以改进的地方

1. 默认参数使用了 `[]`，例如 `cards=[]`、`suits=[]`。  
   Python 中可变默认参数可能造成隐藏状态污染，建议改成 `None` 后在函数内部初始化。

2. 部分模块依赖全局单例 `GameConfig`。  
   这让函数之间耦合较强，测试和调试时不太方便。

3. 对手建模部分基本没有启用。  
   目前主要使用自摸概率 `T_SELFMO`，对其他玩家的弃牌倾向、危险牌判断考虑不足。

4. 评分函数仍然偏规则。  
   可以考虑用深度学习模型替换部分评估模块，例如：
   - 出牌评分修正。
   - 有效进张价值评估。
   - 对手手牌风险估计。
   - 吃碰杠收益判断。

5. 搜索树深度有限。  
   当前更关注从当前组合扩展到胡牌路径，但对多轮对局、其他玩家行为和牌墙不确定性的建模还可以增强。
